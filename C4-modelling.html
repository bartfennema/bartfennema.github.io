Visuals First: self explaining communication power for (Software) Architects using C4 One of the most important responsibilities of a Software Architect is to communicate; to gather, validate and distribute information. Your audiences differ widely; from business people, IT-people, programmers to managers and archiving experts. In my belief, the architect should adapt/translate its language and level of detail to match the particular audience. My mantra is “Visuals First!” because if text processing occurs on one side of the brain, and image processing in another part, lets use both! Practically none of these audiences have studied the language, tools, graphics or schema-language architects often use; like UML, ArchiMate, BPMN and such. You use these tools probably for a good reason, but chances are almost 100%, your audiences do not understand these architecture schemas. That is a problem. If your audience does not understand the architect’s schema, your message also does not get across.

1. C4 to the rescue

Simon Brown created the C4 model to communicate Software Architecture. It originally contains 4 levels of detail;

  • Context
  • Container
  • Component
  • Code

The idea is brilliantly clean and easy, however Simon’s approach being a ‘Coder’ has a focus on the Code side of it.

2. There are a few challenges with that perspective:

  • The Component and Code level is where updates occur too frequently. It often is too much detail for the Solution Architecture purpose.
  • Architecture within the teams requires very skilled teams. In my organisation we grow them, we usually don't have them always available 'on the shelf'.
  • In my world of projects, a certain level of upfront architecture is needed.
  • Simon likes to generate the architecture schema from json or annotations in the source-code. This is where my perspective is different.

3. The benefits of C4

The beauty of the C4 diagrams is in the simplicity, the clarity, the way nearly anyone can interpret the schema without the need for years of IT education. Due to the near-meaningless shapes and very meaningful description and relations the communication power can be enormous!

  • I am able to communicate complex contexts and 'how systems work' with audiences that never understood these before.
  • It shows me what I don’t know (!) and it allows stakeholders who know, to help me out. I even use this on purpose by ‘making mistakes’, triggering the counterparts to start elaborating about all kinds of relevant things.
  • In projects I tend to ‘loose’ these diagrams, and after a while I hear 'good ideas' about them from within the organisation.

I am a big fan of the Context and Container views, complemented with a Deployment view. As a system integrator (we implement solutions built upon standard software platforms), this is my bare minimum. Next to these, I create all kinds of schema’s using the similar notation; how information flows, how processes run, how layered security in a document repository is built up. The notation is where I gain most value.

4. My C4 Notation Rules

The C4 notation works on whiteboards and any low-tech sketching solution, thus anyone in the project can maintain, not only those that master version control. On my schema, every box represents a ‘thing’, like a system, component, part, process, group or person:

  • a name (bold), the name for the ‘thing’ the organisation understands, like ‘MyE-LearningTool’, or ‘ActiveDirectory’.
  • ```A stereotype, between [] or <<>>, usually italic, containing a specific term for the ‘thing’ like [ECM system] or ```
  • A description of the responsibility of the thing, or the expectations of the person, groep or external connected system. It brings enormous value if one can define the responsibilities of the things in the IT landscape or architecture. Often I cannot, being a good indicator I don’t understand the environment enough…

Boxes are connected with directional arrows that always have a description of what traverses that line. This arrow can for example represent an insurance policy, ‘customer data’ or very specific ‘filename1.pdf’.

5. The C4 notation in C4 notation

In the deployment diagram, add to the lines the protocols and port-numbers, this is the data they expect and need. In the boxes I usually add the type/version of OS, running services, if relevant CPU and memory etc.

6. Beware with symbols and colour!

  • Symbols: don’t assume your audience understands the shape of a database; use a legenda
  • Colour: what if you print in black-and-white? Strong suggestion: try it out, use dashes and dash-dot combinations instead (and explain in your legenda!)
  • Add a Title to the schema (as Simon Brown prescribes, be specific!) and I always add a version number and a date. From time to time I update these schemas, publish them on a wiki, print them and put them on the wall. It helps to see if they are outdated.

Once ‘good enough’, these C4 schema’s should be rather stable. Architecture artifacts should not be changed every day, then it contains too much detail. These schemas should be abstract enough to last a while, high level enough to communicate the right message, and specific enough for a particular audience or communication goal. It is not the exact workbook to derive implementation code from, that still needs some thought.

The C4 schemas defined by Simon Brown do not solve all communication challenges. When crafted properly, they are a very, very good start for powerful communication and effective detection if you understand your architecture challenge.
